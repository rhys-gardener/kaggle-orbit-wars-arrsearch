# AGENTS.md

This file provides guidance to Codex (Codex.ai/code) when working with
code in this repository.

## Competition

Kaggle Orbit Wars (deadline 2026-06-23). A 100×100 RTS game: 500 turns,
2–4 players. The submission is `main.py` — specifically the `agent(obs)`
function at module level.

## What this repo is

This is the **array-search track** for Orbit Wars. The bet: training states
are generated by synthesizing initial array states (from `sim.make_env(seed)`
at step 0 only, never `.step()`) and rolling them forward with pure ndarray
operations. **No kaggle env or fast_env in the training hot path.**

The sibling repo `kaggle-orbit-wars-rg` (one level up) holds the
heuristic / evolved / v17-search lineage. That track plateaued mid-leaderboard;
this repo is a clean divergence.

Read [docs/plans/array-search-initiative.md](docs/plans/array-search-initiative.md)
first.

## Architecture

### Submission flow

`main.py` is the only file Kaggle evaluates. It must be self-contained (no
imports from `src/`, no relative imports). It exposes `agent` at module level.
The current `main.py` carries the v17-style scored / embedded-ranker baseline
inherited from the old repo — it works and submits today. The plan is to
replace the agent body with an array-search ranker once trained, embedding
weights into `main.py` the same way the v17 ranker is embedded
(`_v17_embedded_ranker`, `_v17_embedded_gate`).

### Training pipeline (planned)

```
random seed
  └─ sim.make_env(seed) → step-0 obs (never .step())
     └─ initial arrays + geometry cache (closed-form orbit math)
        └─ multi-seat self-play rollout (array_env only)
           └─ records: (graph_state, candidates, action_sets) per seat per turn
              └─ rollout-based labels → train ranker → re-cache → repeat
```

### Key modules

- `src/graph_training/state.py` — graph state extraction (planet/edge
  features, inferred fleet events).
- `src/graph_training/actions.py` — candidate launch generation
  (engine-confirmed or geometry-cache backed).
- `src/graph_training/search.py` — beam search over candidates →
  whole-turn action sets.
- `src/graph_training/array_env.py` — pure-ndarray rollout: scheduling,
  production, combat resolution, scoring.
- `src/graph_training/geometry_cache.py` — closed-form precomputation of
  bucketed `(step, src, tgt, ship_bucket)` geometry.
- `main.py` — submission agent + physics helpers (`fleet_speed`,
  `intercept`, `sun_blocked`, `is_orbiting`, `_solve_engine_angle`,
  `_trajectory_first_hit`) consumed by `src/graph_training/*`.
- `sim/fast_env.py` — `make_env(...)` used only as a source of
  real-distribution initial obs at step 0.

### Key game engine facts

- **Game loop order per turn**: fleet launch → production → fleet
  movement+collision → **planet movement** → combat. Fleet collision uses
  planet position *before* that turn's planet movement → use `int(eta)`
  (floor), not ceil.
- **Fleet fields**: `[id, owner, x, y, angle, from_planet_id, ships]`
  — no `to_planet_id`. Determine target via ray-circle intersection on
  `(x, y, angle)`.
- **Planet orbits**: `x = cx + r*cos(initial_angle + av*step)` using
  `obs.angular_velocity`.
- **Comets**: use `obs.comets[].paths[path_index + eta]` — do NOT apply
  circular-orbit formula to comets.
- **Fleet speed**: `min(6.0, 1.0 + 5.0 * (log(ships)/log(1000))^1.5)` —
  larger fleets are faster.

## Project Rules

- **Always get user approval before submitting to Kaggle.**
- **Submission cap is 5 per 24h.** Don't burn slots speculatively.
- **Run local checks before submitting** — at minimum, run one full game
  via `sim.make_env` to confirm `main.py` doesn't crash and produces moves.
- **Keep `findings.md` up to date.** After every submission, add the score
  to the Submission Log.

## Key Constraints

- **Do not use `uv add kaggle-environments`** — `open-spiel` dependency
  won't build on Windows. Use `uv pip install "kaggle-environments>=1.28.0" --no-deps`.
- **Do not create `.ipynb` files** — use `.py` scripts instead.
- `main.py` must remain self-contained for Kaggle submission; no relative
  imports, no imports from `src/` or `sim/`.
- Training code must stay in `src/` and `scripts/`. The training pipeline
  never imports kaggle_environments and never calls `.step()` on `sim.make_env`.
